## Structure
- `packages/core` — shared interfaces/utils/types. `packages/server` — embedded server + gateway (connections → `src/gateway/connections/`, orchestration → `src/gateway/orchestration/`, Slack OAuth → `src/gateway/routes/public/slack.ts`). `packages/agent-worker` — agent execution (OpenClaw, `src/openclaw/`); talks only to gateway+agent, no platform knowledge.
- **Platform isolation:** InteractionService events carry a `platform` field; each renderer filters on its own identity (`platform === "telegram"`), never another's.
- Bun workspaces under `packages/*`; TS in `packages/*/src`, tests in `__tests__`. Prefer `bun` over `npm`. Fix unused params by deleting them, not `_`-prefixing. Landing copy (`packages/landing`): no em dashes in user-facing text.
- **Submodule `packages/owletto`:** push owletto to a reachable branch BEFORE bumping the pointer (unreachable SHA breaks prod cloning). Before pushing any `bun.lock`/`package.json` change: `git submodule update --init packages/owletto && bun install --frozen-lockfile`; commit any regenerated lockfile in the same change.
- **Frontend (owletto):** before UI changes, read `packages/owletto/DESIGN_GUIDELINES.md` (not auto-loaded) and follow it; don't add new primitives without updating it in the same PR.

## Architecture
- **Platform:** all chat platforms run through Chat SDK adapters in `packages/server/src/gateway/connections/`; connections are created via the `/agents` UI or CRUD API (no per-platform env vars). Webhooks are the default transport — don't add per-platform SDKs/transports. Telegram alone has `mode: auto|webhook|polling`; `polling` is rejected when `LOBU_CLOUD_MODE=1`.
- **Orchestration:** one app process runs gateway + worker orchestrator + embeddings + memory. Workers are `child_process.spawn` subprocesses (`cwd=./workspaces/{agentId}/`, `WORKSPACE_DIR` env; Linux wraps them in `systemd-run --user --scope` for limits + `IPAddressDeny`). Postgres (pgvector; optional postgis) is the only external, via `DATABASE_URL`; runtime state lives in PG tables. Workers NEVER see real creds — the gateway `secret-proxy` swaps `lobu_secret_<uuid>` placeholders at egress.
- **🚨 Multi-replica is prod reality — every change MUST be correct under N>1 app replicas.** k8s Deployment (`charts/lobu`); chart default is `app.replicaCount:1` / `sessionAffinity:None`, but prod opts into `replicaCount:2+` and `service.sessionAffinity:ClientIP`. Per-pod state (`SseManager` + its event backlog, the in-process `workers` map, the deploy-lock cache) is in-memory and pod-local with no cross-pod fan-out. Cross-replica delivery rides Postgres (`thread_response` queue). API/SSE terminal rows and interaction cards are owner-routed: a pod without the client's SSE connection throws to re-queue until the owning pod claims (`unified-thread-consumer.ts` owner-gate, `TERMINAL_DELIVERY_SEND_OPTS` 30×1s). Headless rows (`platformMetadata.source` ∈ watcher-run/connector-repair/scheduled-job/internal — no SSE client exists on any pod) are exempt and delivered on first claim. Streaming deltas/status stay best-effort: claimed on the wrong pod they're lost (accepted; cross-pod fan-out is a planned follow-up). **Chat connections are rows, not processes:** no boot warm-start; every replica hydrates a connection on demand from its `agent_connections` row (instance = memo keyed on `updated_at`, so config edits converge lazily — never assume a pod has a warm instance, and never add cross-pod restart fan-out). Exclusive transports (Telegram polling) run on exactly ONE replica via the `connection_claims` lease (15s tick / 45s TTL); webhook transports must stay runnable on any replica. Status health is owned by the single-claimant `connection-health` task, not boot side-effects. **Never put shared state in an in-memory Map/singleton another replica must read/mutate.** Before claiming a feature works, answer: *"does this hold with 3 replicas behind ClientIP affinity?"* — if it relies on one component seeing another's in-memory event across pods, it's broken; use a Postgres-mediated signal.
- **MCP:** providers from `config/providers.json`; MCP servers from per-agent settings or `SKILL.md`. Workers discover MCP tools at startup and call them via the gateway proxy (JWT). Built-ins (snake_case, registered in `packages/agent-worker/src/openclaw/custom-tools.ts`): `ask_user`, `upload_file`, `generate_image`, `generate_audio`, `list_conversations`, `read_conversation`, `send_message` (past channel conversation is read via `search_memory`, which also scans `channel_messages`). Integration auth (OAuth/refresh) lives in Lobu MCP servers — workers never see tokens.
- **`events` is append-only.** Never `DELETE FROM events`; tombstone via `client.knowledge.delete()` / `save_knowledge({ supersedes_event_id, ... })`. `current_event_records` masks superseded rows.
- **Connectors:** `*.connector.ts` extending `ConnectorRuntime`. npm deps → project `package.json`, bundled by esbuild at compile time; native deps → `runtime.nix.packages` (nixpkgs refs), provisioned via `nix-shell` at run. Compile happens on the CLI (`lobu apply` → `bun install --ignore-scripts` if bun is on PATH, else `npm install --ignore-scripts` — Node ships npm so bun is optional on the user's machine); `@lobu/connector-sdk` is externalized (runtime-provided).
- **Guardrails** (`packages/core/src/guardrails/`): stages `input`/`output`/`pre-tool`; built-ins `secret-scan`, `pii-scan`, `forbidden-tools` (see `gateway/guardrails/builtins.ts` + `aggregator.ts`). Configure via `defineAgent({ guardrails: [...] })` in `lobu.config.ts`; all fail open on infra errors; each trip writes a `guardrail-trip` event.
- **Network/egress:** worker subprocesses get `HTTP_PROXY=http://localhost:8118`. Access via `WORKER_ALLOWED_DOMAINS`: empty=no net (default), `github.com`=allowlist, `*`=all (not prod), `*`+`WORKER_DISALLOWED_DOMAINS`=blocklist; domains exact or `.wildcard`. Linux prod adds kernel `IPAddressDeny=any` + allow 127.0.0.1. Risky domains can route through an LLM egress judge (Haiku, 5-min cache, fail-closed; needs `ANTHROPIC_API_KEY`; `gateway/proxy/http-proxy.ts`).

## Releasing
release-please owns it: land conventional commits on `main`, merge the generated release PR, CI publishes via OIDC (see `docs/RELEASING.md`). Inter-package deps MUST be `"@lobu/<name>": "workspace:*"`. Don't hand-edit `packages/*/package.json` versions or push `chore(release)` commits.

## Agent rules
- **Never write to `~/Code/lobu` directly — always work in a worktree.** `make task-setup NAME=<slug>` first; main stays on `main` so others can `git worktree add`. Submodule-pointer-only bump: `make bump SUBMODULE=<path> [TARGET=<ref>]`.
- Do only what's asked. Don't create `*.md` unless asked. Delete ephemeral files you create. After editing `packages/agent-worker/*`, run `make clean-workers`.
- Slack link pasted (`slack.com/archives/…?thread_ts=`) → run `scripts/slack-thread-viewer.js "<link>"` first. Unsure in planning → `codex exec "Q" --config model_reasoning_effort="high"`.
- **No new dynamic imports** outside the allow-list; default to static `import`. A new `await import(...)` needs a measured cost justification added here + a rationale comment at the call site. Allow-list: cli `index.ts`, `commands/_lib/connector-run-cmd.ts`, `commands/_lib/apply/apply-cmd.ts`, `commands/_lib/apply/desired-state.ts`, `commands/dev.ts`, `commands/memory/browser-auth.ts`; server `server.ts` (server-lifecycle), `embedded-runtime.ts` (embedded-postgres/pgvector), `sandbox/run-script.ts` (loads the per-Node-major `isolated-vm` / `isolated-vm-next` native optional build — no static equivalent since only one build exists per host, and the module may be absent; cost is one import of an already-on-disk native addon); connector-worker `executor/child-runner.ts` + `self-check/index.ts` (load a freshly-compiled connector bundle from a runtime-computed temp path, and probe `@lobu/connector-sdk`/`@lobu/core` resolution at runtime — operations with no static equivalent; cost is one import of an already-on-disk bundle, identical to the production execution path). Tests: `await import` in `beforeAll`/`beforeEach`/`test()` is allowed (post-`vi.mock`).

## Branch hygiene
- One branch = one concern, but bundle related work (fewer, larger, reviewable PRs). Tangential ask mid-branch: commit+push current, open the PR, branch fresh off `main` (stack only if it builds on unmerged code).
- **Never `git stash`** — use WIP commits, squash later. Parallel sessions → `claude --worktree <name>`. Subagents that may switch/commit/push/destroy MUST run `isolation: "worktree"`; read-only research can share the parent.
- **Cross-repo:** owletto changes go through `make task-setup` (real branch, not detached HEAD); bump the lobu submodule pointer as a separate PR after the owletto merge. Don't pass absolute repo paths in dispatch prompts.

## Development & validation
- Prereqs: Bun, Node **22–24 or 26+** (only Node 25 is unsupported), reachable Postgres+pgvector via `DATABASE_URL`. Optional postgis (`./scripts/setup-local-postgis.sh`) for reverse-geocoding. The SDK sandbox ships two `isolated-vm` builds side by side via `optionalDependencies`: `isolated-vm@6` (Node 22–24) and the aliased `isolated-vm-next` = `isolated-vm@7` (Node 26+). `loadIsolatedVm` (`run-script.ts`) picks per Node major; `assert-node-version.ts` boots on any Node ≥22 and only warns on 25, where neither build exists (an EOL odd-numbered line upstream skipped). So no host is rejected for the sandbox except 25, and bumping the prod/CI baseline (Docker still Node 24) is now decoupled from this. Engine field is `>=22 <25 || >=26`.
- `./scripts/setup-dev.sh` (first time; provisions brew `postgresql@18` on :5418). `make dev` is **local dev against the shared brew Postgres@18** (one DB per branch via `dev-db.sh`, same major as the product's embedded PG18) + workers + Vite on :8787 — a single long-lived postmaster avoids the embedded per-worktree clusters whose kill-9 churn leaked SysV shm anchors into macOS's `shmmni=32`. `LOBU_EMBEDDED=1 make dev` (≡ `make dev-embedded`) runs the zero-dependency embedded per-worktree cluster — still the `lobu run` / CI runtime, just no longer the dev default. `make clean-workers` (orphans). Parallel worktrees: gitignored `.env.local` with non-default `PORT`/`WORKER_PROXY_PORT` (seeded by `task-setup`, read by `dev-db.sh`). Don't `git switch` while a dev server runs. Husky pre-commit runs `biome check --write`.
- **After changes run `make review`** (typecheck/unit/integration + pi verdict; posts a PR comment). Build per change: landing → `cd packages/landing && bun run build`; `{core,server,agent-worker,cli}` → `make build-packages`; broad → `bun run typecheck` (`make dev` doesn't rebuild workspace pkgs).
- **E2E before merge (hard gate)** for bug fixes: red→fix→green reproducer in the PR body; if you can't reproduce, BAIL (post the dead-end, don't open a PR). Native-app UI exempt (compile-check + draft, say so).
- Bot behavior → `./scripts/test-bot.sh "@me ..."` (dev `@clawdotfreebot`, prod `@lobuaibot`; clear stale history via `chat_state_lists`). Prompt/behavior → promptfoo (`cd examples/personal-finance && bun run evals`). Auth'd UI → `docs/BROWSER_TESTING.md`.
- **Owletto e2e (Chrome/Mac):** Chrome → `make e2e-browser` launches/reuses the stable `owletto` agent-browser harness (persistent `~/.config/lobu-dev/chrome` profile + `--extension` from this worktree); drive it with `agent-browser --session owletto <cmd>`, `RESTART=1` to pick up extension/worktree changes. The extension's fixed manifest `key` pins its ID, so chrome.storage.local (gateway URL + auth) persists — pair once, reuse across sessions/worktrees, like the installed Mac app. Mac → use the installed `Owletto.app`; it reads `~/.config/lobu/config.json` each popover. `make task-setup` does **not** register a Lobu CLI context by default — pass `CONTEXT=1` when you want that worktree in the menubar picker (`lobu context rm` / `make task-clean` removes it). Don't reintroduce a per-worktree extension symlink.
- `.env` is the single source of truth for secrets (restart `make dev` after changes). Worker sessions persist via `./workspaces/{agentId}/`. Skills declare network/nix needs that merge into the allowlist — review skills before installing; destructive MCP calls need in-thread approval unless pre-approved in `lobu.config.ts`.
