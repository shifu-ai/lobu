Design brainstorm

Workflows & the WAIT primitive

A Behavior fires once. A Workflow needs to do → wait → do. The only genuinely-new piece is WAIT — and it turns out approval is a wait too. Everything else rides behaviors + events + a scheduler you already have.

1 · The one new primitive

A Workflow = a Behavior whose action is a sequence with WAIT steps — not a new engine, just a richer body. Simple behavior = one step. Workflow = steps + waits. The list still shows one sentence; you expand to see the flow.

Behavior (today)

When X → do Y

One event, one action. Fires and finishes.

The gap

…then wait…

Suspend until a deadline / a count / an event / a human. Collect while waiting.

Workflow

do → wait → do

A behavior with a multi-step body. WAIT is the only new node.

3 · Four kinds of WAIT (one primitive, four conditions)

Wait

Duration / deadline

"20 minutes" · "until 12:00". Backed by a scheduler row that resumes the run.

Wait

Count / threshold

"until 5 replies" · "until quorum". A cheap check on the accumulating state each time an event lands.

Wait

A specific event

"until the PR merges" · "until payment clears". Correlate an inbound event to the run.

Wait

A human

= approval

"until I approve". This is the approval gate — same primitive, human as the resume condition.

The unlock: approval isn't a separate feature — it's Wait-for-human. So the WAIT primitive we build for workflows and the "needs approval" governance step are one mechanism: suspend the run, park state, resume on a signal (a clock, a count, an event, or a person clicking approve).

4 · How it rides the substrate (no new engine)

PieceMaps to what exists
Workflow defAn ordered list of steps + waits, stored on the behavior. (Behavior body = array instead of a single action.)
Run stateA workflow_run row (or a keyed Surface via supersedes_event_id) holding: current step, accumulated events, resume condition.
CollectInbound events (replies) correlated to the run and appended — the same event ingest you already have, tagged with run_id.
The wait itselfA scheduled_jobs row for the deadline (wake_agent, exists + grantable today) + a cheap threshold/event check on each new correlated event. Whichever fires first resumes.
ResumeThe resume signal wakes the run and invokes the agent for the next step, with the collected state in context.
ApprovalSame as any wait — the resume signal is a human clicking approve in the inbox (the governance gate).

Only genuinely-new state: a suspended run that holds partial state and a resume condition. Everything else — events, scheduler, surfaces, agent turns — already exists.

5 · How to represent a workflow — brainstorm the shape

A · Multi-step Behavior lean

A behavior can have a multi-step body (do/wait/do). Simple ones stay one step. Same tab, same list, expand to edit the flow.

+ One model, no new tab; simple stays simple
– Editor must handle both simple + flow modes
B · Dedicated Workflow builder

A separate "Workflows" surface with a visual flow canvas (nodes + branches).

+ Room for branching / complex flows
– New tab, new metaphor; Zapier-drawer risk; overkill for do→wait→do
C · Chained behaviors

Behavior A completes → emits an event → Behavior B listens for it. No explicit workflow object.

+ Zero new concepts
– State + wait scattered across behaviors; no single thing to debug/toggle
✓ LOCKED: Option A — a workflow is a Behavior with a longer body + WAIT nodes. One sentence, expand for detail; no second builder. Reach for B only if real branching shows up. Time‑waits work today (`manage_schedules(wake_agent)` is grantable, enabled by default for workflow agents). Event‑waits ride the Watch detector via a resume‑subscription (§below). Watch is near‑real‑time (cron + incremental + skip‑if‑empty batching) — real‑time reserved for chat.

6 · Open questions to brainstorm

Where does partial state show while suspended?
In Runs as a "waiting" row (like approval), expandable to the live collection? Or as a temporary Surface? Probably: Runs row for status + the run holds the state.
Timeouts & the empty case
Every wait needs a max (a deadline is built-in; a pure count-wait needs a fallback timeout). On resume with empty collection → step decides (skip / post "nothing"). Never hang forever.
Cancellation & re-entrancy
Can you cancel a suspended run? What if the trigger fires again while one is still waiting (two lunch polls)? Likely: one active run per workflow, new fire is skipped or queued.
Is "collect" its own step, or a property of WAIT?
Lean: a property of WAIT ("wait, and while waiting accumulate matching events into state"). Keeps the primitive count low.
Multi-replica safety
The suspended run + resume must be claimable by exactly one replica (the scheduler already does this for scheduled_jobs — reuse that leasing).
